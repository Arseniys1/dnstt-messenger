# Требования к интеграции ботов

## Введение

Данный документ описывает требования к системе интеграции ботов в мессенджер с E2E шифрованием, работающий через DNS-туннель. Боты должны функционировать как программно управляемые пользователи, способные получать и отправлять зашифрованные сообщения, работать с комнатами и реагировать на команды.

## Глоссарий

- **Bot**: Программно управляемый пользователь мессенджера, работающий через Bot_API
- **Bot_API**: Серверный API для создания и управления ботами
- **Bot_SDK**: Клиентская библиотека на Go для разработки ботов
- **Bot_Token**: Уникальный токен аутентификации бота (UUID формат)
- **Bot_Handler**: Функция обработчик событий в боте
- **Server**: TCP-сервер мессенджера на Go
- **Client**: Клиентское приложение мессенджера
- **E2E_Message**: Сообщение, зашифрованное end-to-end с использованием X25519 + ChaCha20-Poly1305
- **DM**: Личное сообщение (Direct Message) между двумя пользователями
- **Room**: Групповая комната для обмена сообщениями
- **Command**: Текстовое сообщение, начинающееся с префикса "/" (например, /help)
- **Bot_Process**: Отдельный процесс операционной системы, в котором выполняется бот

## Требования

### Требование 1: Регистрация и управление ботами

**User Story:** Как администратор сервера, я хочу создавать и управлять ботами через API, чтобы интегрировать автоматизированные сервисы в мессенджер.

#### Критерии приемки

1. THE Server SHALL предоставлять HTTP API endpoint для создания бота с параметрами (имя, описание)
2. WHEN администратор создает бота, THE Server SHALL генерировать уникальный Bot_Token и сохранять его в базе данных
3. THE Server SHALL возвращать Bot_Token в ответе на запрос создания бота
4. THE Server SHALL хранить информацию о ботах в таблице базы данных с полями (id, login, token_hash, description, created_at, is_active)
5. THE Server SHALL предоставлять HTTP API endpoint для получения списка всех ботов
6. THE Server SHALL предоставлять HTTP API endpoint для деактивации бота по его login
7. WHEN бот деактивирован, THE Server SHALL отклонять попытки аутентификации с его Bot_Token

### Требование 2: Аутентификация ботов

**User Story:** Как разработчик бота, я хочу аутентифицировать бота с помощью токена, чтобы безопасно подключиться к серверу.

#### Критерии приемки

1. THE Server SHALL принимать новую команду CmdBotLogin (0x24) с payload [TokenLen(1)][Token(N)]
2. WHEN Server получает CmdBotLogin, THE Server SHALL проверять Bot_Token в базе данных
3. IF Bot_Token валиден и бот активен, THEN THE Server SHALL отправлять CmdLoginOK с назначенным SID
4. IF Bot_Token невалиден или бот неактивен, THEN THE Server SHALL отправлять CmdLoginFail
5. THE Server SHALL хранить хеш Bot_Token (bcrypt) вместо открытого значения
6. WHEN бот успешно аутентифицирован, THE Server SHALL обрабатывать его как обычного пользователя для всех операций обмена сообщениями

### Требование 3: Bot SDK - Подключение и жизненный цикл

**User Story:** Как разработчик бота, я хочу использовать простую Go библиотеку для подключения бота к серверу, чтобы не реализовывать низкоуровневый протокол вручную.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять функцию NewBot(config BotConfig) для создания экземпляра бота
2. THE Bot_SDK SHALL принимать конфигурацию с полями (ServerAddr, ProxyAddr, DirectMode, BotToken, MaxFrameSize)
3. THE Bot_SDK SHALL предоставлять метод Connect() для установки соединения с сервером
4. WHEN метод Connect() вызван, THE Bot_SDK SHALL выполнять ECDH handshake и отправлять CmdBotLogin
5. THE Bot_SDK SHALL предоставлять метод Disconnect() для корректного закрытия соединения
6. THE Bot_SDK SHALL автоматически загружать или генерировать E2E ключи бота при инициализации
7. THE Bot_SDK SHALL предоставлять метод Run() для запуска цикла обработки входящих сообщений

### Требование 4: Bot SDK - Получение сообщений

**User Story:** Как разработчик бота, я хочу регистрировать обработчики для входящих сообщений, чтобы реагировать на события в мессенджере.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять метод OnE2EMessage(handler func(msg E2EMessage)) для регистрации обработчика E2E сообщений
2. THE Bot_SDK SHALL предоставлять метод OnDM(handler func(msg DMMessage)) для регистрации обработчика личных сообщений
3. THE Bot_SDK SHALL предоставлять метод OnRoomMessage(handler func(msg RoomMessage)) для регистрации обработчика сообщений в комнатах
4. WHEN бот получает зашифрованное сообщение, THE Bot_SDK SHALL автоматически расшифровывать его с использованием E2E ключей
5. THE Bot_SDK SHALL передавать расшифрованное сообщение в соответствующий Bot_Handler с метаданными (отправитель, timestamp, roomID для комнат)
6. IF расшифровка сообщения не удалась, THEN THE Bot_SDK SHALL логировать ошибку и пропускать сообщение

### Требование 5: Bot SDK - Отправка сообщений

**User Story:** Как разработчик бота, я хочу отправлять сообщения через простые методы SDK, чтобы бот мог отвечать пользователям.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять метод SendE2EMessage(text string) для отправки E2E сообщений всем онлайн пользователям
2. THE Bot_SDK SHALL предоставлять метод SendDM(recipient string, text string) для отправки личного сообщения
3. THE Bot_SDK SHALL предоставлять метод SendRoomMessage(roomID uint32, text string) для отправки сообщения в комнату
4. WHEN метод отправки вызван, THE Bot_SDK SHALL автоматически шифровать сообщение для всех получателей
5. THE Bot_SDK SHALL автоматически фрагментировать сообщения, превышающие MaxFrameSize
6. THE Bot_SDK SHALL запрашивать публичные ключи получателей, если они отсутствуют в кеше
7. IF публичный ключ получателя недоступен, THEN THE Bot_SDK SHALL возвращать ошибку

### Требование 6: Bot SDK - Работа с комнатами

**User Story:** Как разработчик бота, я хочу управлять участием бота в комнатах, чтобы бот мог работать в групповых чатах.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять метод JoinRoom(roomID uint32) для входа бота в комнату
2. THE Bot_SDK SHALL предоставлять метод LeaveRoom(roomID uint32) для выхода бота из комнаты
3. THE Bot_SDK SHALL предоставлять метод GetRoomList() для получения списка доступных комнат
4. THE Bot_SDK SHALL предоставлять метод GetRoomMembers(roomID uint32) для получения списка участников комнаты
5. WHEN бот входит в комнату, THE Bot_SDK SHALL автоматически получать историю сообщений комнаты
6. THE Bot_SDK SHALL предоставлять метод OnRoomJoined(handler func(roomID uint32)) для уведомления о входе в комнату

### Требование 7: Bot SDK - Обработка команд

**User Story:** Как разработчик бота, я хочу легко обрабатывать команды пользователей, чтобы создавать интерактивных ботов.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять метод OnCommand(command string, handler func(ctx CommandContext)) для регистрации обработчика команды
2. WHEN бот получает сообщение, начинающееся с "/", THE Bot_SDK SHALL парсить команду и аргументы
3. THE Bot_SDK SHALL вызывать соответствующий Bot_Handler с контекстом CommandContext
4. THE CommandContext SHALL содержать поля (Command, Args []string, Sender, RoomID, IsDirectMessage)
5. THE Bot_SDK SHALL предоставлять метод RegisterDefaultHandler(handler func(msg Message)) для обработки сообщений без зарегистрированного обработчика
6. IF команда не зарегистрирована, THEN THE Bot_SDK SHALL вызывать RegisterDefaultHandler если он установлен

### Требование 8: Bot SDK - Управление состоянием

**User Story:** Как разработчик бота, я хочу получать информацию о состоянии подключения и онлайн пользователях, чтобы бот мог адаптировать свое поведение.

#### Критерии приемки

1. THE Bot_SDK SHALL предоставлять метод GetOnlineUsers() для получения списка онлайн пользователей
2. THE Bot_SDK SHALL автоматически обновлять внутренний список онлайн пользователей при получении CmdOnlineList, CmdOnlineAdd, CmdOnlineRemove
3. THE Bot_SDK SHALL предоставлять метод IsConnected() для проверки состояния подключения
4. THE Bot_SDK SHALL предоставлять метод OnUserOnline(handler func(username string)) для уведомления о появлении пользователя онлайн
5. THE Bot_SDK SHALL предоставлять метод OnUserOffline(handler func(username string)) для уведомления об отключении пользователя
6. THE Bot_SDK SHALL предоставлять метод GetBotLogin() для получения имени бота

### Требование 9: Запуск ботов как отдельных процессов

**User Story:** Как администратор, я хочу запускать ботов как независимые процессы, чтобы изолировать их выполнение и управлять ими отдельно.

#### Критерии приемки

1. THE Bot_SDK SHALL поддерживать компиляцию бота в отдельный исполняемый файл
2. THE Bot_SDK SHALL предоставлять возможность загрузки конфигурации из JSON файла
3. THE Bot_SDK SHALL предоставлять возможность передачи Bot_Token через переменную окружения BOT_TOKEN
4. THE Bot_SDK SHALL логировать события бота в stdout с префиксом [BOT]
5. WHEN Bot_Process завершается с ошибкой, THE Bot_SDK SHALL возвращать ненулевой exit code
6. THE Bot_SDK SHALL корректно обрабатывать сигналы SIGINT и SIGTERM для graceful shutdown

### Требование 10: Безопасность ботов

**User Story:** Как администратор сервера, я хочу контролировать доступ ботов к функциям мессенджера, чтобы предотвратить злоупотребления.

#### Критерии приемки

1. THE Server SHALL хранить Bot_Token в хешированном виде (bcrypt) в базе данных
2. THE Server SHALL генерировать Bot_Token с энтропией не менее 128 бит (UUID v4)
3. THE Server SHALL ограничивать количество попыток аутентификации бота до 3 попыток в минуту с одного IP
4. THE Server SHALL логировать все попытки аутентификации ботов с указанием IP адреса
5. THE Server SHALL предоставлять возможность установки rate limit для отправки сообщений ботом (по умолчанию 10 сообщений в секунду)
6. IF бот превышает rate limit, THEN THE Server SHALL отклонять сообщения и отправлять ошибку боту
7. THE Server SHALL предоставлять HTTP API endpoint для ротации Bot_Token существующего бота

### Требование 11: Примеры и документация

**User Story:** Как разработчик бота, я хочу иметь примеры и документацию, чтобы быстро начать разработку.

#### Критерии приемки

1. THE Bot_SDK SHALL включать пример простого echo-бота в директории examples/echo-bot
2. THE Bot_SDK SHALL включать пример бота с обработкой команд в директории examples/command-bot
3. THE Bot_SDK SHALL включать README.md с инструкциями по установке и быстрому старту
4. THE Bot_SDK SHALL включать документацию API в формате godoc для всех публичных методов
5. THE Bot_SDK SHALL включать пример конфигурационного файла bot_config.json
6. THE Server SHALL включать документацию HTTP API для управления ботами в формате OpenAPI/Swagger

### Требование 12: Совместимость с существующей архитектурой

**User Story:** Как разработчик системы, я хочу интегрировать ботов без изменения существующего протокола клиентов, чтобы сохранить обратную совместимость.

#### Критерии приемки

1. THE Server SHALL обрабатывать ботов как обычных пользователей после успешной аутентификации
2. THE Server SHALL использовать существующие команды (CmdE2EMsg, CmdDM, CmdRoomMsg) для обмена сообщениями с ботами
3. THE Server SHALL сохранять сообщения ботов в существующих таблицах (messages, dm_messages, room_messages)
4. THE Server SHALL добавлять новую таблицу bots без изменения схемы существующих таблиц
5. THE Server SHALL поддерживать федерацию для ботов аналогично обычным пользователям
6. WHEN обычный Client получает сообщение от бота, THE Client SHALL отображать его как сообщение от обычного пользователя
